Resumen completo del proyecto "Quibdó Seguro"

Objetivo del documento

Este archivo resume la estructura, funcionalidades, flujo de autenticación, microservicios y configuraciones principales del proyecto "Quibdó Seguro" para que puedas enviarlo a ChatGPT (u otra IA) y obtener ayuda, preguntas o revisiones.

1) Visión general de la aplicación

- Nombre: Quibdó Seguro
- Tipo: Aplicación web Laravel con arquitectura híbrida (monolito UI + microservicios).
- Propósito: Permitir a usuarios reportar incidentes, ver un mapa, ganar puntos/recompensas, y a comercios/autoridades gestionar incidentes y recompensas.
- Componentes principales: UI monolítica (raíz), microservicios (auth-service, gateway, incident-service, notification-service, rewards-service, analytics-service).

2) Arquitectura y despliegue local

- Monolito (UI raíz): corre en http://localhost:8006 (aplicación Laravel que sirve vistas, formularios y actúa como cliente hacia el gateway).
- Gateway: corre en http://localhost:8000 y reenvía/actúa como proxy hacia los microservicios.
- auth-service: corre en http://localhost:8001 y gestiona registro, login, logout, perfil, tokens.
- Otros servicios (pueden usar puertos 8002..8005): incident-service, notification-service, rewards-service, analytics-service.

3) Bases de datos

- Cada microservicio incluye su propio `config/database.php` y puede conectarse a su propia base de datos.
- En este proyecto varios servicios usan MongoDB (ej.: `auth-service`, `incident-service`).
- El gateway está configurado con su propio `config/database.php` (aunque sirve principalmente como proxy).

4) Autenticación y tokens

- Auth: implementado por `auth-service` usando Laravel Sanctum con un modelo `PersonalAccessToken` adaptado a MongoDB.
- Formato de token: cuando se crea un token la respuesta incluye `id|plain` (ej.: `6a150783...|WXAWB9sB...`).
- Flujo web (monolito):
  - Formularios (login/register) envían datos a `AuthController` del monolito.
  - `AuthController` hace HTTP POST a `GATEWAY_URL` (ej. `http://localhost:8000/auth/login` o `/auth/register`).
  - Si la petición es desde navegador: guardamos el token devuelto en cookie `quibdo_token`, ejecutamos `Auth::login($user)` localmente y redirigimos a `/dashboard`.
  - Logout: `AuthController::logout()` extrae el token del header Bearer o de la cookie `quibdo_token`, lo envía a `gateway` → `auth-service` para revocarlo, ejecuta `Auth::logout()` en el monolito y borra la cookie.
- Notas especiales:
  - Se creó un `PersonalAccessToken` compatible con MongoDB y se registró con `Sanctum::usePersonalAccessTokenModel(...)`.
  - Para evitar colisiones de throttling durante el proxy, se deshabilitaron o ajustaron middlewares `ThrottleRequests` en algunos kernels (gateway y auth-service) y se implementó un control de rate propio.

5) Rutas y endpoints principales

- Monolito (web):
  - `GET /` → inicio
  - `GET /login` → vista login (formulario) — `AuthController@showLogin`
  - `POST /login` → `AuthController@login` (proxied to gateway)
  - `GET /register` → vista registro
  - `POST /register` → `AuthController@register`
  - `POST /logout` → `AuthController@logout` (envía token a gateway)
  - `GET /dashboard`, `/perfil`, `/reportar`, `/mapa`, `/recompensas`, etc. — protegidas con `auth` middleware
  - Rutas de admin y comercio con middleware `role` y `auth`.

- Gateway (API routes):
  - `POST /auth/login` → proxy hacia `auth-service`
  - `POST /auth/register` → proxy hacia `auth-service`
  - `POST /auth/logout` → proxy hacia `auth-service`
  - El gateway también reenvía `Authorization: Bearer <token>` y multipart/form-data cuando aplica.

- Auth-service (API):
  - `POST /api/auth/login`
  - `POST /api/auth/register`
  - `POST /api/auth/logout` (requiere token)
  - `GET /api/auth/profile` (requiere token)
  - Endpoints internos `GET /api/auth/user/{id}`, `/users-by-role/{rol}`, `PATCH /api/auth/user/{id}/puntos` para otros servicios.

6) Vistas y UX

- `resources/views/login.blade.php`: formulario de login con manejo de errores y CSRF.
- `resources/views/register.blade.php`: formulario de registro con confirmación de contraseña.
- Al inicio se mostró JSON crudo cuando los controladores actuaban puramente como proxy; se cambió para:
  - Diferenciar entre peticiones API/AJAX (devuelven JSON) y peticiones web normales (redirigen y usan cookies/flash messages).

7) Archivos y cambios importantes que implementamos

- `app/Http/Controllers/AuthController.php` (monolito):
  - Proxy a gateway para login/register, pero para peticiones web guarda cookie `quibdo_token`, inicia sesión local y redirige.
  - `logout()` ahora extrae token de header Bearer o cookie `quibdo_token`, llama al gateway para revocar token y borra cookie.
- `microservices/gateway/*`: proxy endpoints y configuración de rate limiting.
- `microservices/auth-service/app/Models/PersonalAccessToken.php`: modelo personalizado para almacenar tokens en MongoDB (extiende `Laravel\Sanctum\PersonalAccessToken` y usa `DocumentModel`).
- `microservices/auth-service/app/Providers/AuthServiceProvider.php`: registro del provider `mongodb` y `Sanctum::usePersonalAccessTokenModel(...)`.
- `microservices/auth-service/app/Services/AuthService.php`: lógica de registro/login/logout/profile; incluye `userFromRequest()` para resolver token manualmente cuando el middleware no lo hace.
- Algunos `config/auth.php` y `app/Http/Kernel.php` fueron ajustados temporalmente para evitar conflicto de `ThrottleRequests` y guard recursivos.

8) Variables de entorno y configuración clave

- En el monolito y servicios:
  - `GATEWAY_URL` (ej. `http://localhost:8000`)
  - `DB_CONNECTION` / `MONGODB_URI` / `DB_DATABASE` para servicios que usan MongoDB
  - `LOG_CHANNEL`, `SESSION_DRIVER` (asegúrate que estén configurados correctamente)
  - Puertos locales: gateway 8000, auth-service 8001, monolito 8006 (por convención en este repo)

9) Cómo probar manualmente (pasos probados)

- Registro via direct service:
  - curl -X POST http://localhost:8001/api/auth/register -d '{"nombre":"X","email":"x@example.com","password":"secret","password_confirmation":"secret"}'
- Registro via gateway:
  - curl -X POST http://localhost:8000/auth/register -d '...'
- Login via gateway:
  - curl -X POST http://localhost:8000/auth/login -d '{"email":"...","password":"..."}'
- Web login (simular navegador):
  - GET `/login` para obtener cookie `XSRF-TOKEN` y `laravel_session`
  - POST `/login` con header `X-XSRF-TOKEN` y campos `email`, `password` → la UI ahora redirige y guarda `quibdo_token` en cookie
- Logout:
  - POST `/logout` en el monolito (usa token de cookie o Authorization header) → revoca token en auth-service y borra cookie

10) Problemas encontrados y soluciones aplicadas

- Problema: 429 Too Many Requests - causado por `ThrottleRequests` en gateway y servicios durante proxy.
  - Solución: ajustar/deshabilitar `throttle:api` en kernels donde hacía conflicto.
- Problema: Sanctum intentaba usar modelo SQL y SQL driver al crear tokens (error `prepare()` en null).
  - Solución: crear `PersonalAccessToken` que extiende la clase de Sanctum y usar `Sanctum::usePersonalAccessTokenModel()`.
- Problema: Logout devolvía `Unauthenticated.` porque middleware `auth:sanctum` rechazaba la petición antes de resolver token.
  - Solución: fallback de resolución manual (`userFromRequest`) y, en el monolito, enviar token desde cookie.

11) Recomendaciones y siguientes pasos

- Reforzar el forwarding del header `Authorization` en el gateway para todos los proxied requests.
- Revisar la política de CORS y `SANCTUM_STATEFUL_DOMAINS` si se usan cookies cross-domain en producción.
- Rehabilitar `auth:sanctum` middleware en rutas protegidas una vez que el `PersonalAccessToken` y la configuración del guard estén totalmente validados, para mantener compatibilidad con las herramientas de Laravel.
- Añadir tests automatizados (Feature tests) que verifiquen registro/login/logout a través de gateway.
- Documentar la lista completa de puertos y comandos `php artisan serve` o `docker-compose` si planeas desplegar en contenedor.
