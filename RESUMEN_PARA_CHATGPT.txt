RESUMEN DEL PROYECTO: Quibdó Seguro

Objetivo general
- UI monolítica (Laravel) mantenida como frontend mientras el backend está dividido en microservicios (gateway + varios servicios).
- La intención es que la página web funcione igual que antes, pero que todas las operaciones backend (auth, incidentes, notificaciones, recompensas, analytics) sean provistas por microservicios detrás de un gateway.

Estructura general del repositorio
- Raíz: vistas, controladores web y rutas principales (puerta de enlace web en puerto 8006 en desarrollo).
- Carpeta `microservices/`: servicios independientes:
  - `gateway/` (proxy central, puerto 8000) — recibe peticiones y las reenvía a servicios específicos.
  - `auth-service/` (autenticación, puerto 8001) — registro, login, logout, profile, manejo de tokens Sanctum con MongoDB.
  - `incident-service/` (reportes/incidentes, puerto 8002)
  - `notification-service/` (notificaciones, puerto 8003)
  - `rewards-service/` (recompensas, puerto 8004)
  - `analytics-service/` (analíticas, puerto 8005)

Puntos clave de autenticación
- Flujo web:
  - Formularios `GET /login` y `GET /register` en la UI monolítica.
  - `POST /login` y `POST /register` en la UI delegan al gateway: `GATEWAY_URL + /auth/login` o `/auth/register`.
  - Respuesta del gateway incluye `data.token` (formato: `<id>|<plain>`).
  - La UI guarda el token en cookie `quibdo_token` (30 días) y además ejecuta `Auth::login($user)` localmente para usar las middlewares `auth` de Laravel.
  - Logout: la UI toma token de `Authorization` o de cookie `quibdo_token` y llama al gateway `/auth/logout` para revocar el token en `auth-service`, luego borra la cookie y hace `Auth::logout()`.

- Token storage y Sanctum:
  - `auth-service` usa un modelo `PersonalAccessToken` adaptado para MongoDB y se registra con `Sanctum::usePersonalAccessTokenModel(...)`.
  - Tokens se guardan en la colección `personal_access_tokens` en MongoDB. El valor almacenado en `token` es el hash (sha256) del token en texto plano.
  - `auth-service` implementa una función `userFromRequest` para localizar `tokenable` desde `findToken()` cuando el middleware `auth:sanctum` no se emplea o no resuelve al usuario (fallback).

Rutas importantes (resumen)
- Monolito (web):
  - `/` — inicio
  - `/login` (GET/POST) — vista + proxy a gateway
  - `/register` (GET/POST)
  - `/logout` (POST) — proxy a gateway y limpieza local
  - `/dashboard`, `/perfil`, `/reportar`, `/mapa`, `/recompensas`, etc. — rutas protegidas con `auth` (Laravel web session)
- Gateway (`microservices/gateway/routes/api.php`):
  - `/auth/*` — reenvía a `auth-service`
  - Redirecciones y proxy multipart/json gestionadas en closures del gateway
- Auth-service (`microservices/auth-service/routes/api.php`):
  - `POST /api/auth/register`, `POST /api/auth/login`, `POST /api/auth/logout`, `GET /api/auth/profile` (y endpoints internos `user/{id}`, `users-by-role/{rol}`)

Bases de datos
- Cada microservicio tiene su propio `config/database.php` y puede usar su propia base (independencia):
  - `auth-service` y `incident-service` están configurados para MongoDB.
  - `gateway` tiene su propio `config/database.php` (por si necesita persistencia local, pero en el flujo actual actúa como proxy).

Archivos y ubicaciones clave
- Frontend / monolito:
  - Controlador de auth (proxy): `app/Http/Controllers/AuthController.php`
  - Vistas: `resources/views/login.blade.php`, `resources/views/register.blade.php`
  - Rutas web: `routes/web.php`
- Gateway:
  - `microservices/gateway/routes/api.php`
  - Kernel y middleware en `microservices/gateway/app/Http/Kernel.php`
- Auth-service:
  - Rutas: `microservices/auth-service/routes/api.php`
  - Controller: `microservices/auth-service/app/Http/Controllers/AuthController.php`
  - Servicio: `microservices/auth-service/app/Services/AuthService.php`
  - PersonalAccessToken (Mongo model): `microservices/auth-service/app/Models/PersonalAccessToken.php`
  - Provider: `microservices/auth-service/app/Providers/AuthServiceProvider.php`
  - Config DB: `microservices/auth-service/config/database.php`

Comportamiento ya implementado (lo que pediste)
- Login/register desde la UI usan el gateway y crean tokens en auth-service.
- Web no muestra JSON crudo: formulario redirige y guarda cookie `quibdo_token`.
- Logout revoca token en `auth-service`, borra cookie y cierra sesión local.
- Fallback para autenticación por token: `auth-service` puede resolver usuario desde token cuando middleware no lo hace.

Cómo probar rápidamente (ejemplos)
- Registro directo al auth-service (JSON):
  curl -X POST http://localhost:8001/api/auth/register -H 'Content-Type: application/json' -d '{"nombre":"Test","email":"t@example.com","password":"secret","password_confirmation":"secret"}'
- Registro vía gateway (JSON):
  curl -X POST http://localhost:8000/auth/register -H 'Content-Type: application/json' -d '{...}'
- Login web (simular navegador con CSRF):
  1) GET http://localhost:8006/login  (obtener cookie XSRF-TOKEN)
  2) POST form a http://localhost:8006/login con header `X-XSRF-TOKEN: <valor>` y fields `email` y `password`.
- Probar profile con token:
  curl -H "Authorization: Bearer <id|plain>" http://localhost:8001/api/auth/profile

Notas de implementación y problemas resueltos
- Se eliminó o deshabilitó el `throttle:api` predeterminado en gateway y `auth-service` para evitar 429 en pruebas locales de proxy.
- Se creó un modelo `PersonalAccessToken` compatible con MongoDB y se registró en Sanctum para evitar errores tipo/incompatibilidades.
- Se añadió lógica para resolver usuario por token manualmente cuando `auth:sanctum` no resuelve (esto facilita el gateway/proxy).
