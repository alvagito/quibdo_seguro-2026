DOCUMENTACIÓN TÉCNICA — Quibdó Seguro

Objetivo
- Proveer a ChatGPT (o a cualquier revisor) una descripción técnica completa del proyecto: arquitectura, metodología de desarrollo, estimación temporal, decisiones de infraestructura (Docker/Kubernetes), lista final de microservicios y recomendaciones prácticas.

1) Metodologías de desarrollo (opciones)
- Scrum
  - Iteraciones (sprints) de 1-2 semanas.
  - Roles: Product Owner, Scrum Master, Equipo de desarrollo.
  - Beneficio: entrega incremental, adaptación rápida a cambios.
- Cascada (Waterfall)
  - Fases secuenciales: requisitos → análisis → diseño → implementación → pruebas → despliegue.
  - Beneficio: si requisitos son estables y bien definidos.
- XP (Extreme Programming)
  - Desarrollo orientado a prácticas: TDD, pair programming, integración continua, entregas frecuentes.
  - Beneficio: calidad de código y feedback continuo.
- Otra (híbrida)
  - Recomendado: enfoque ágil híbrido (Scrum + prácticas XP) para este proyecto: sprints cortos + revisión continua + CI/CD.

Recomendación: usar Scrum con prácticas XP (TDD, CI, code review). Esto encaja bien con microservicios y despliegues frecuentes.

2) Estimación temporal (ejemplo de planificación)
(Asumiendo equipo pequeño 2–4 devs, alcance basado en estado actual del repo)
- Análisis y requisitos: 1 semana
- Diseño de arquitectura y APIs (contratos entre servicios): 1 semana
- Desarrollo inicial (microservicios core + gateway): 3 semanas
  - Auth, Incident, Rewards, Notification, Gateway endpoints básicos
- Pruebas e integración (incluye tests e2e): 1 semana
- Harden/producción (Docker, CI/CD, monitoreo): 1 semana
Total estimado: ~7 semanas

Desglose alternativo por fases (más detallado)
- Semana 1: análisis detallado + definir contratos (OpenAPI/Swagger) + DB schemas
- Semana 2: diseño y scaffolding (docker-compose, common libs, gateway)
- Semanas 3-5: desarrollo de servicios (paralelizar por servicio)
- Semana 6: pruebas integradas, correcciones y seguridad
- Semana 7: despliegue, documentación, handoff

3) ¿Usarán Docker o Kubernetes?
- Recomendación práctica:
  - Desarrollo local: Docker (docker-compose) para levantar todos los servicios (gateway, auth, incident, mongo, etc.).
  - Producción: Kubernetes (cluster) si se espera alta disponibilidad o escalado; alternativamente usar servicios PaaS o ECS si se prefiere menos operaciones.
- Estado actual del repo: Laravel + MongoDB + microservicios. No hay docker-compose incluido por defecto (a menos que exista en otra rama). Se recomienda crear `docker-compose.yml` para desarrollo y preparar manifests helm/K8s para producción.

4) Microservicios finales (confirmación)
Según tu estructura y lo solicitado, la lista final de microservicios debe ser:
- Auth Service — gestión de usuarios, tokens (Sanctum adaptado a MongoDB)
- Incident Service — reportes, consultas de incidentes, mapa
- Rewards Service — catálogo de recompensas, canjes
- Notification Service — envío y gestión de notificaciones
- Analytics Service — métricas, agregaciones, dashboards
- API Gateway — proxy central, rutas, forwarding, rate-limiting

Cada servicio puede tener su propia base de datos (independencia): auth-service y incident-service ya están configurados para MongoDB en el repo.

5) Configuración y variables críticas
- `GATEWAY_URL` (monolito y servicios que llaman al gateway)
- `MONGODB_URI` / `DB_DATABASE` (por servicio)
- `SANCTUM` config y `personal_access_token_model` (auth-service)
- `SESSION_DRIVER`, `LOG_CHANNEL` en cada app

  - Monolito (UI): 8006

6) Recomendaciones de despliegue y entorno
- Implementar `docker-compose.yml` para desarrollo con servicios: auth, incident, notification, rewards, analytics, gateway, mongo, redis (si aplica), nginx (opcional).
- En producción: empaquetar cada microservicio en su propia imagen Docker y usar Kubernetes con:
  - Deployments, Services, Ingress (TLS), ConfigMaps/Secrets para variables.
  - Horizontal Pod Autoscaler (HPA) para servicios con alta carga.
- Seguridad de tokens: marcar cookie `quibdo_token` como `Secure`, `HttpOnly`, `SameSite=Strict` en producción y usar HTTPS.
- Protección de API: rate limiting en gateway, logging/correlation ID, autenticación mutua para comunicación interna si es necesario.

7) Tests y Calidad
- Unit tests por servicio (PHPUnit / Pest para Laravel).
- Contract tests o OpenAPI-based tests para asegurar compatibilidad entre gateway y servicios.
- End-to-end tests (simular flujo web: register → login → profile → logout).
- Integración continua: GitHub Actions / GitLab CI: build, test, lint, image push.

8) Observabilidad y operación
- Logs centralizados (ELK/EFK) o usar un servicio de logging.
- Tracing (OpenTelemetry) para seguimiento de requests a través del gateway entre servicios.
- Métricas: Prometheus + Grafana.
